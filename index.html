Python Binary Search Template
def binary_search_on_answer(low, high, condition_check):
    ans = -1
    while low <= high:
        mid = low + (high - low) // 2
        if condition_check(mid):
            ans = mid
            # Try to optimize further depending on min/max
            high = mid - 1
        else:
            low = mid + 1
    return ans
🖥️ Core CS Concepts Summary (OS & DBMS)
📥 Download Detailed PDF Guide

Welcome to the OS & DBMS tutorial block! Let's break down the complex terms into simple lessons as if we are studying in a classroom. Let's dive in.

Operating Systems (OS)
  • Process vs Thread: Imagine a Process as an independent factory building with its own workspace, and Threads as the individual workers sharing that same space. Since workers share the floor, they can talk instantly, but they might crash into each other (race conditions) without proper coordination (locks).
  • CPU Scheduling: The scheduler acts as a gatekeeper. FIFO handles processes in order. SJF handles the shortest task first (optimal, but requires predicting the future!). Round Robin gives everyone a small time slice (time quantum) in turns.
  • Deadlocks: This is like two cars meeting head-on on a one-lane bridge; neither can move forward or backward. Remember the four conditions: Mutual Exclusion, Hold & Wait, No Preemption, and Circular Wait.
  • Virtual Memory: When your RAM is full, the OS borrows space from the hard drive (Paging). Thrashing occurs when the OS spends all its time swapping pages in/out of storage instead of actually running your program!
Database Management Systems (DBMS)
  • SQL Joins:
    • Inner Join: Matches rows present in both tables.
    • Left Join: All rows from left table + matched rows from right.
    • Right Join: All rows from right table + matched rows from left.
    • Full Join: All rows when there is a match in either table.
  • ACID Properties: Think of ACID as the database safety net: Atomicity (all or nothing changes), Consistency (no broken rules), Isolation (independent runs), and Durability (saved changes remain even if the power cuts!).
  • Indexing: An index is like the index at the back of a textbook. It lets you find topics (rows) instantly, but writing new pages (inserts/updates) takes slightly longer because you have to update the index list too.

🏛️ High-Level & Low-Level Design Basics
📥 Download Detailed PDF Guide

Welcome class! System design is where we transition from 'writing code' to 'building architectures'. Let's explore how we scale systems to handle millions of students.

High-Level Design (HLD)
  • Horizontal vs Vertical Scaling: Think of Vertical Scaling as buying a bigger, faster truck (more RAM/CPU) to carry packages. It works up to a limit, but if the truck breaks down, everything stops! Horizontal Scaling is like hiring a fleet of 50 smaller delivery vans. If one breaks down, the others carry the load.
  • Load Balancers: The dispatcher of your fleet. It takes incoming customer orders (traffic) and assigns them to active vans (servers) so no single driver gets overwhelmed.
  • Caching: Like keeping your favorite books on your desk instead of walking to the public library every time you need to read a page. We store hot database records in memory (Redis) to return them in milliseconds.
  • API Gateway: A security guard at the front gate. It checks guest IDs, limits how many people enter per minute (rate limiting), and routes them to the correct building (microservices).
Low-Level Design (LLD)
  • SOLID Principles: A blueprint for clean code:
    • Single Responsibility: One tool, one job (a hammer only hits nails).
    • Open/Closed: Plug-in modules (add features by adding new modules, don't re-wire the base system!).
    • Liskov Substitution: Child components must fit perfectly where parent components go.
    • Interface Segregation: Smaller custom panels (don't force a user to use a remote control with 100 unused buttons!).
    • Dependency Inversion: Plug into sockets (your lamp depends on the electrical outlet abstraction, not the concrete power grid!).
  • Common Design Patterns: Singleton (only one database connection manager), Factory (a factory that manufactures different types of payment cards), Observer (subscribers getting newsletters automatically when published).

🍃 MongoDB Cheatsheet
📥 Download PDF Cheatsheet
1. Database Commands
show dbs               // View all databases
use dbName             // Create or switch database
db                     // View current database
db.dropDatabase()      // Delete current database
2. Collection Commands
show collections       // Show collections
db.createCollection('comments')  // Create collection
db.comments.drop()     // Drop collection
3. Document Operations (CRUD)
db.comments.find()     // Find all documents
db.comments.find().pretty() // Find all with pretty spacing
db.comments.findOne({name: 'Harry'}) // Find matching item
db.comments.insertOne({
  name: 'Harry',
  lang: 'JavaScript',
  member_since: 5
})
db.comments.insertMany([
  {name: 'Harry', lang: 'JavaScript', member_since: 5},
  {name: 'Rohan', lang: 'Python', member_since: 3}
])
4. Query Operators
db.comments.find({member_since: {$lt: 90}})  // Less than
db.comments.find({member_since: {$gte: 90}}) // Greater than or equal
db.comments.find({$or: [{lang: 'Python'}, {lang: 'Java'}]}) // Logical OR
db.comments.find({lang: {$in: ['Python', 'Java']}}) // In array
5. Indexes & Aggregations
db.comments.createIndex({name: 1}) // Create index
db.comments.aggregate([
  {$group: {_id: "$lang", total: {$sum: 1}}} // Count grouped by field
])

Ultimate Hackathon Guide

1. Pre-Hackathon Prep (48 – 24 hrs before)
  • Choose track early: Rank event challenges by domain interest, sponsor influence, and judging criteria weight.
  • Brainstorm 2-3 solid ideas per track: Use ChatGPT / Perplexity to riff 20+ prompts, then converge on top 3 with problem -> solution -> impact statements.
  • Validate with sponsors: Pitch your shortlist at the booths. Build whichever concept excites the actual sponsor judge.
  • Read API docs: Bookmark endpoints, authentication flow codes, and quota limits.
2. Assemble a Balanced Team
  • Product Lead/Pitcher: Idea refinement, rubric mapping, demo setup.
  • Frontend Dev: UI/UX design (Next.js + Tailwind + Vercel v0).
  • Backend/DevOps: API endpoints, DB schema, deployments (Supabase/Vercel).
  • ML/AI Wrapper: LLM prompt integration (OpenAI API / Gemini).
3. Lightning-Fast Build Workflow (0 – 12 hrs)
  1. Scaffold UI: Prompt v0, download resulting UI mockup as a zip file.
  2. Let Cursor digest: Open folder in Cursor. Wait 5 minutes to let it index, then run queries.
  3. Backend in one shot: Define DB schema in Supabase SQL editor. Ask Cursor to "Create Express routes for /signup, /login, /transactions using Supabase client".
  4. Integrate Sponsor APIs: Keep to one flagship integration that demos well. Target MVP deployment in ≤ 5 hours.
4. Leveraging AI (Prompt Engineering 101)

Prompt engineering is the craft of framing and refining the inputs you give an AI model to get precise, useful outputs.

  • Assign a clear role ("You are an expert TypeScript engineer").
  • State your goal ("Generate an Express route to fetch user data").
  • Define desired output formats ("Return only valid code, no commentary").
  • Use few-shot examples or chain-of-thought prompts.
5. Nail the Presentation (Last 2 hrs)
  • Map slides to rubric: If Innovation counts for 30%, have a dedicated Innovation slide.
  • Hook the room: Open with a 15-second story or audience live poll.
  • Live Demo: Login -> Key Feature -> Wow moment (≤ 90 seconds). Have a backup pre-recorded Loom.
